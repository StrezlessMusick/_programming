6.43.1 Basic Asm — Assembler Instructions with No Operands

The asm keyword allows you to embed assembler instructions within C code. 

     asm [ volatile ] ( AssemblerInstructions )

To create headers compatible with ISO C, write __asm__ instead of asm (see Alternate Keywords).

By definition, a Basic asm statement is one with no operands. asm statements that contain one or more colons (used to delineate operands) are considered to be Extended (for example, asm("int $3") is Basic, and asm("int $3" : ) is Extended). See Extended Asm.

Qualifiers

volatile
This optional qualifier has no effect. All Basic asm blocks are implicitly volatile.

Parameters

AssemblerInstructions
This is a literal string that specifies the assembler code. The string can contain any instructions recognized by the assembler, including directives. GCC does not parse the assembler instructions themselves and does not know what they mean or even whether they are valid assembler input. The compiler copies it verbatim to the assembly language output file, without processing dialects or any of the "%" operators that are available with Extended asm. This results in minor differences between Basic asm strings and Extended asm templates. For example, to refer to registers you might use %%eax in Extended asm and %eax in Basic asm.

You may place multiple assembler instructions together in a single asm string, separated by the characters normally used in assembly code for the system. A combination that works in most places is a newline to break the line, plus a tab character (written as "\n\t"). Some assemblers allow semicolons as a line separator. However, note that some assembler dialects use semicolons to start a comment.

Do not expect a sequence of asm statements to remain perfectly consecutive after compilation. If certain instructions need to remain consecutive in the output, put them in a single multi-instruction asm statement. Note that GCC's optimizers can move asm statements relative to other code, including across jumps.

asm statements may not perform jumps into other asm statements. GCC does not know about these jumps, and therefore cannot take account of them when deciding how to optimize. Jumps from asm to C labels are only supported in Extended asm.

Remarks

Using Extended asm will typically produce smaller, safer, and more efficient code, and in most cases it is a better solution. When writing inline assembly language outside of C functions, however, you must use Basic asm. Extended asm statements have to be inside a C function. Functions declared with the naked attribute also require Basic asm (see Function Attributes).

Under certain circumstances, GCC may duplicate (or remove duplicates of) your assembly code when optimizing. This can lead to unexpected duplicate symbol errors during compilation if your assembly code defines symbols or labels.

Safely accessing C data and calling functions from Basic asm is more complex than it may appear. To access C data, it is better to use Extended asm.

Since GCC does not parse the AssemblerInstructions, it has no visibility of any symbols it references. This may result in GCC discarding those symbols as unreferenced.

Unlike Extended asm, all Basic asm blocks are implicitly volatile. See Volatile. Similarly, Basic asm blocks are not treated as though they used a "memory" clobber (see Clobbers).

All Basic asm blocks use the assembler dialect specified by the -masm command-line option. Basic asm provides no mechanism to provide different assembler strings for different dialects.

Here is an example of Basic asm for i386: 

     /* Note that this code will not compile with -masm=intel */
     #define DebugBreak() asm("int $3")
